Which architecture approach fits best?

All options use Next.js App Router + Mantine + Supabase. The difference is how data flows.

A

Full Server-First (Server Components + Server Actions)

All data fetching happens in Server Components. Mutations use Server Actions. Supabase runs entirely on the server. Mantine UI components are wrapped in "use client" boundaries only where needed.

Pros

  • Most secure — Supabase keys never exposed to browser
  • Best performance — data arrives with the HTML
  • No loading spinners for initial page load

Cons

  • Complex mental model — "use client" boundary management
  • Mantine requires careful client/server split
  • Harder to implement real-time interactivity
B

Hybrid — SSR for pages, client for interactions ⭐ Recommended

Server Components fetch initial data (questions list, session config). Client Components handle interactive UI (study session, forms). Uses @supabase/ssr for both server and client Supabase instances.

Pros

  • Best of both worlds — fast initial loads + smooth interactions
  • Supabase's officially recommended pattern for Next.js
  • Mantine works naturally on the client side
  • Auth handled server-side (protected routes are secure)

Cons

  • Need to manage two Supabase client instances (server + browser)
  • Slightly more setup upfront
C

Client-Side SPA style

Most pages are Client Components using Supabase JS directly, similar to how the previous project used Apollo Client. Auth is handled via Supabase's auth state on the client.

Pros

  • Simplest mental model — closest to the previous project's pattern
  • Mantine works everywhere without configuration
  • Fastest to build

Cons

  • No SSR benefits — pages load blank then fill in
  • Auth checking happens client-side (brief flash before redirect)
  • Wastes Next.js's main strengths
